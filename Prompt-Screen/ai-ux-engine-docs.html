1. System Overview

AI UX Engine generates deterministic UX flows and Figma-ready UI frames using domain knowledge as the source of truth. AI is optional and only used for enrichment. The current system is knowledge-driven; a design graph layer is planned for the next version.

Input
Prompt + domain knowledge.
Process
Resolve domain, select flow, build layouts.
Output
JSON frames rendered to Figma.

2. Knowledge Models

Domain knowledge is stored in `config/domainModels.json`. Each model contains the entity name, keywords, texts, and fields. This ensures consistent UI labels, CTA copy, and form structure.

{
  "student": {
    "entity": "Student",
    "keywords": ["student", "enrollment"],
    "texts": {
      "ctaText": "Enroll Student",
      "headerText": "Students"
    },
    "fields": [
      {"label": "Student Name", "type": "text"},
      {"label": "Email", "type": "email"}
    ]
  }
}

3. Domain Resolver

`engine/domainResolver.js` selects the best domain model using exact matches first, then keyword scoring. If no match exists, it falls back to the last meaningful token.

domain = resolveDomain(prompt)
// domain.source: "domain-models" | "prompt"
// domain.matchType: "exact" | "keyword" | "fallback"

4. AI Mode Rules

`engine/runModeResolver.js` decides when AI is used. AI is skipped for exact domain matches or explicit exact prompts. This keeps behavior deterministic for known domains.

useAI = shouldUseAI(prompt, domain)
if (domain.matchType === "exact") useAI = false

5. Flow Generator

`ai/flowGenerator.js` chooses between Flow A (table) and Flow B (dashboard) based on prompt keywords like `dashboard`, `grid`, or `app`.

flow = generateFlow(prompt, domain)
// Flow A: table → form → filledForm → table → table → details
// Flow B: dashboard → form → filledForm → dashboard → dashboard → details

6. Field Resolver

`engine/fieldResolver` pulls fields from the domain model. If AI is enabled, it can fill missing fields or context.

fields = resolveFields(prompt, domain, { step, useAI })

7. Component Mapper

`engine/componentMapper.js` maps field types to UI components (text, email, dropdown, etc.).

mappedFields = mapFields(fields)

8. Slot System

Slots are named placeholders that control UI structure. Modules attach to slots and content fills them.

slot.module.*
slot.content.*
slot.overlay.*

9. Layout Builder

`builder/layoutBuilder.js` converts flow steps and mapped fields into deterministic layouts with slots injected.

layout = buildLayout(step, mappedFields)

10. JSON Builder

`builder/jsonBuilder.js` transforms layouts into renderer-ready frames.

frames = buildJson(layouts)

11. Figma Renderer

`renderer/figmaRenderer.js` emits JSON to Figma or returns JSON when in preview mode.

return figmaRenderer(frames, { mode: "json" })

12. Knowledge Pipeline

These scripts generate the knowledge base from Figma and tokens:

  • `scripts/fetchFigma.js`
  • `scripts/extractDesignData.js`
  • `scripts/generateKnowledge.js`
  • `scripts/schema/generateWithIconsSchema.js`

13. Output Artifacts

Knowledge and schema outputs are written to:

  • `knowledge/components/knw-components.json`
  • `knowledge/components/knw-group-component.json`
  • `knowledge/components/knw-icons.json`
  • `knowledge/tokens/knw-design-tokens.json`
  • `knowledge/patterns/knw-ux-patterns.json`
  • `knowledge/schemas/generated-form-stages.schema.json`

14. Prompt Examples

"Student management dashboard"
Matches `student`, Flow B.
"Create a visitor log"
Matches `visitor`, Flow A.
"Order tracking form"
Matches `order`, Flow A.
"Application dashboard"
Exact prompt → AI off.

15. Next Version (Design Graph)

The next version introduces a design graph layer to make flow selection and UX branching explicit. The current knowledge pipeline already provides the data foundation; the graph will orchestrate it across multi-path flows.
